# Implementation Plan

> **前提条件**: 本実装は ai-settings-model-picker(feat/186192-ai-settings-model-picker)のマージ後に着手する(design.md「Revalidation Triggers」参照 — 同一ファイル群を変更するため)。

- [ ] 1. 基盤: モデル識別子と共有型
- [ ] 1.1 モデル識別子(modelKey)ユーティリティを実装する
  - プロバイダとモデル ID から複合キーを生成し、最初のセパレータで分割して復元する pure 関数群(往復不変。不正な prefix・空のモデル ID・セパレータ不在は null)
  - クライアント供給値向けの防御的な長さ上限定数を旧上限から移管して宣言する
  - 完了条件: 生成・解析の往復、モデル ID 内セパレータの許容、不正値の null 返却を検証するユニットテストがグリーン
  - _Requirements: 2.1_
- [ ] 1.2 プロバイダ設定・許可モデル・API 契約の共有型を再定義する
  - プロバイダごとの設定(有効フラグ・Azure 接続設定)とプロバイダ別 API キー集合の型を新設する
  - 許可モデルに所属プロバイダ(必須)を追加し、許可判定述語を(プロバイダ, モデル ID)の組で照合する形に変更する
  - 管理 API の応答/更新 DTO(4 プロバイダの Record 形・キー設定済みフラグ)、チャット API の DTO(provider 情報付きモデル一覧 + 選択キー)、管理対象 config キー一覧を再定義する
  - 完了条件: interfaces 層の型・述語のユニットテストがグリーン(下流層の追従は後続タスクで実施)
  - _Requirements: 1.2, 2.1_

- [ ] 2. 基盤: config スキーマと env 構成
- [ ] 2.1 config スキーマと env-only 部分ロックを再構成する
  - 複数プロバイダ設定(非秘匿)とプロバイダ別 API キー(秘匿)の 2 キーを JSON env var 対応で新設し、旧単一プロバイダ 3 キーを削除する(自動移行なし)
  - env-only グループの対象を「AI 有効トグル + プロバイダ接続設定 2 キー」に差し替え、許可モデルを対象から外す
  - 完了条件: config 定義の spec がグリーンで、旧キー参照が config 定義から消えている。新形式未設定時(旧設定のみ残存時)は AI 未設定扱いになる
  - _Requirements: 1.1, 1.2, 5.1, 5.2, 5.3, 7.1, 7.2, 7.3_
- [ ] 2.2 環境変数の記述例ドキュメントを整備する
  - 3 つの JSON env var の具体例(JSON エスケープ、複数プロバイダ・複数キーの合成、Azure の endpoint/Entra ID 例、既定モデル指定)を作成する
  - 完了条件: `.kiro/specs/mastra-multi-provider/docs/env-configuration-examples.md` が存在し、8.2 のスモークテスト手順がこの記述例をそのまま使って検証する
  - _Requirements: 5.1_

- [ ] 3. コア: サーバ側の可用性判定とモデル解決
- [ ] 3.1 プロバイダ別 config アクセサと防御ガードを実装する
  - API キー・プロバイダ設定・許可モデルの読取をプロバイダ引数付きに変更する(キー不在エラーはプロバイダ名と env var 名のみ。キー値を含めない)
  - 不正形状の設定値は「未設定」として fail-soft しつつ、(config キー, 理由)の warn を dedup 付きで出力する(dedup ユーティリティは本タスクが所有し、後続タスクが再利用する)
  - 同一 config キーで DB 値と env 値が同時に定義されている場合、env 値がシャドーされている旨を dedup 付き info で出力する(値そのものは出力しない — design「シャドーイングの規則」参照)
  - 完了条件: アクセサの防御ガード・fail-soft warn・シャドーイング info・エラーメッセージ非漏えいのユニットテストがグリーン
  - _Requirements: 1.9, 5.1_
- [ ] 3.2 プロバイダ可用性判定を新設する
  - 「有効(トグル ON)かつ構成済み(必須接続設定あり。Azure は endpoint 必須・Entra ID 時はキー免除)」の単一述語と、有効なプロバイダ/許可モデル集合の導出を実装する
  - 有効なのに構成不備のプロバイダは理由付き warn(3.1 の dedup ユーティリティを再利用。設定更新時のリセット関数を提供)
  - 完了条件: 有効フラグ × キー有無 × Azure 接続のマトリクスと warn の dedup/リセット挙動のユニットテストがグリーン
  - _Requirements: 1.6, 1.7, 6.1_
- [ ] 3.3 実効モデルキー解決を実装する
  - 実効既定モデル(既定エントリが無効なら有効エントリの先頭へ決定的フォールバック)を実装する
  - リクエスト時の単一検証点(有効集合外・省略は実効既定へ丸め + warn、有効モデル 0 件は throw)を実装する
  - config アクセサ(3.1)とは別モジュールとして実装し、アクセサ側からは import させない(config アクセサ → 可用性判定 → 実効キー解決 の一方向依存を保つ — design の依存方向参照)
  - 完了条件: 丸め・フォールバック・0 件 throw のユニットテストがグリーン
  - _Requirements: 4.6, 6.4_
- [ ] 3.4 resolver 群とモデル解決キャッシュを modelKey ベースに変更する
  - 4 プロバイダの resolver が各自の資格情報・接続設定(Azure 含む)を読む形へ変更する
  - modelKey での解決(解析 → ディスパッチ)とキャッシュ、リクエストコンテキスト・エージェントのキー受け渡し、モデル別 provider オプションの(プロバイダ, モデル ID)照合を実装する
  - 完了条件: modelKey ディスパッチ・キャッシュ再構築・オプション照合のユニットテストがグリーン
  - _Requirements: 1.10, 2.8, 4.3_
- [ ] 3.5 (P) AI 有効判定を多プロバイダ意味論に再定義する
  - 「有効なプロバイダが 1 つ以上 かつ その配下の許可モデルが 1 つ以上」で構成済みと判定する(全滅時は未設定 = チャット無効・アプリ継続。旧設定のみ残存時も未設定)
  - 完了条件: 判定マトリクスのユニットテストがグリーン(既存 501 ガードの経路は変更しない)
  - _Requirements: 6.2, 6.3, 7.3_
  - _Boundary: is-ai-configured_
  - _Depends: 3.2_

- [ ] 4.3 設定更新 API を複数プロバイダ対応にする
  - providers を含むリクエストは 4 プロバイダ全エントリ必須(validator)。プロバイダ単位の独立永続化、API キーのみ merge 例外(空/省略 = 維持、消去なし)、無効化時も設定・資格情報・モデル設定を保持する
  - API キー merge の「現在値」は保存処理内で、マージ後ビュー(DB ?? env)から読み取る(GET 時点のスナップショットから再構成しない・env 由来で稼働中のキーを UI 保存で失わせない — design「並行更新の扱い」「PUT セマンティクス」参照)
  - 非空の API キーを 1 つも含まないリクエストではプロバイダ別 API キーの config キーを書き込まない(トグル・モデルのみの保存で env 運用中のキーを DB に複製・固定化しない)
  - トップレベルの各区画(AI トグル・providers・許可モデル)は省略 = 変更なし、存在 = 全量置換とする。許可モデルの空配列は「許可モデルなし」として空配列のまま保存する(キー削除による env への暗黙リセットはしない)
  - env-only 時は許可モデルのみ受理(providers/AI トグルを含む場合 400)。検証は通常時と共通。未構成プロバイダへのモデル保存は許容する
  - 保存成功時に解決キャッシュと不備ログ dedup をリセットする(再起動なし反映)
  - 完了条件: 独立更新・merge 例外(連続 PUT で他プロバイダのキーが保持されること、非空キーなしの PUT でキーの DB 値が作られないこと、env 由来キーがマージ後ビューから引き継がれることを含む)・省略 = 変更なし/空配列 = 空のまま保存・4 エントリ必須・env-only 分割・キャッシュ/dedup リセットの統合テストがグリーン
  - _Requirements: 1.2, 1.3, 1.4, 1.5, 1.6, 1.9, 2.9, 3.3, 5.2, 5.3, 5.4_
