# Requirements Document

## Project Description (Input)

GROWI AI において、同時に複数の Provider, Model を登録できる機能。

### 背景・目的(既存 spec・コードベースからの補足)

- **問題を抱える人**: (a) 用途に応じて異なる LLM ベンダーのモデルを使い分けたい GROWI のエンドユーザー、(b) 複数ベンダーの接続設定と利用可能モデルを一元的に統制したい、セルフホスティング GROWI の管理者・運用者。
- **現状**: [multi-llm-provider](../multi-llm-provider/) により LLM プロバイダは OpenAI / Anthropic / Google / Azure OpenAI から選択できるが、**1 App = 単一プロバイダ**(単一の `ai:provider` / `ai:apiKey` / `ai:azureOpenaiSettings`)に固定されている。[mastra-multi-model-chat](../mastra-multi-model-chat/)(実装済み)により、許可モデル集合 `ai:allowedModels` に複数モデルを登録しエンドユーザーがチャット画面でモデルを選択できるが、選択できるのは**その単一プロバイダ配下のモデルに限られる**。複数プロバイダの同時利用は両 spec とも明示的にスコープ外とされてきた。
- **変えること**: 複数のプロバイダ(それぞれの接続設定・API キーを含む)と、各プロバイダ配下のモデルを**同時に登録**できるようにし、許可モデル集合をプロバイダ横断で構成できるようにする。

### 確定済みの要件判断(ユーザー確認済み)

1. **各プロバイダ種は最大 1 登録**(OpenAI / Anthropic / Google / Azure OpenAI をそれぞれ高々 1 つ。同一プロバイダ種の多重登録はスコープ外)。
2. **既定モデルはグローバルに 1 つ**(プロバイダ横断の許可モデル集合全体から 1 つ。現行の意味論を踏襲)。
3. **一部プロバイダの設定不備は部分縮退**(不備プロバイダのモデルのみ除外し、ログ出力のうえ他プロバイダで継続)。
4. **既存の単一プロバイダ設定からの自動移行は提供しない**(プレリリース前提の置換。旧設定項目は廃止)。
5. **env-only モードは部分ロック**: `AI_USES_ONLY_ENV_VARS_FOR_SOME_OPTIONS=true` のとき、プロバイダの接続設定(API キー等の資格情報・接続先設定)は環境変数のみで構成し管理画面では読み取り専用とする一方、許可モデルの設定(モデル・provider オプション・既定モデル)は管理画面から引き続き自由に変更できる。
6. **設定の保持形式は再設計可**: 現行の config の持ち方(キー構造)の温存は要件としない。複数プロバイダに最適な保持形式へ変更してよい(具体形は設計フェーズで決定)。
7. **プロバイダ構造は固定スロット + 有効/無効トグル**(UI モック準拠): 対応 4 プロバイダを常に設定領域として提供し、動的な追加・削除は設けない。管理者は各プロバイダを有効/無効で切り替え、無効化されたプロバイダのモデルはチャット選択肢から除外される(接続設定・資格情報・モデル設定は保持)。
8. **保存済み API キーの消去操作は提供しない**(キーは上書きのみ。使わないプロバイダは無効トグルで運用)。
9. **管理画面 UI は Claude Design モックを参照**: [ui-design/](./ui-design/) の `AI Settings Multi-Provider.dc.html` / `ProviderPanel.dc.html`(2026-07-02 取り込み)。

### 関連 spec

- [multi-llm-provider](../multi-llm-provider/) — プロバイダ選択(1 App = 1 ベンダー)の導入
- [mastra-multi-model-chat](../mastra-multi-model-chat/) — 同一プロバイダ内の複数許可モデル + チャットでのモデル選択(実装済み)
- [ai-settings-model-picker](../ai-settings-model-picker/) — 管理画面の許可モデル入力をオフライン同梱カタログからの選択式にする(**実装済み**。2026-07-03 に dev/8.0.x へマージされ、本ブランチのベースに含まれる。最終形は同梱資産に加え opt-in のカタログリフレッシュ機構を持つ)

## Introduction

本機能は、GROWI の AI 機能(Mastra チャット)におけるプロバイダ構成を「1 App = 単一プロバイダ」から「1 App = 複数プロバイダの同時利用」へ拡張する。管理者は対応プロバイダ(OpenAI / Anthropic / Google / Azure OpenAI)それぞれに用意された固定の設定領域で資格情報・接続設定・有効/無効を構成し(動的な追加・削除は行わない)、許可モデル集合を複数プロバイダ配下のモデルで横断的に構成できる。エンドユーザーはチャット画面で、プロバイダをまたいだ許可モデルの中からモデルを選択できる。

既定モデルは許可モデル集合全体でちょうど 1 つ(グローバル)とする。一部プロバイダに設定不備がある場合は、そのプロバイダに属するモデルのみを無効化して残りで継続する(部分縮退)。旧来の単一プロバイダ設定は本機能で置き換え、自動移行は提供しない。

本書で「構成済みプロバイダ」とは必須の接続設定(資格情報等)が揃っているプロバイダを、「有効なプロバイダ」とは管理者が有効に切り替えており(トグル ON)かつ構成済みであるプロバイダを指す。

## Boundary Context

- **In scope(本仕様で扱う)**
  - 対応プロバイダ(各種 1 つの固定設定領域)の接続設定・資格情報・有効/無効の構成(管理画面および環境変数)
  - 許可モデル集合のプロバイダ横断化(各許可モデルの所属プロバイダの明示、プロバイダ間での同名モデル識別子の共存)
  - グローバルに 1 つの既定モデル
  - チャット画面でのプロバイダ横断のモデル選択・選択の永続化・サーバ側検証
  - 一部プロバイダの設定不備時の部分縮退と、全プロバイダ不備時の AI 機能無効化
  - env-only モードの部分ロック化(プロバイダ接続設定は環境変数のみ・読み取り専用、許可モデル設定は管理画面から編集可)
  - 旧来の単一プロバイダ設定の廃止(置換、自動移行なし。保持形式の再設計を含む)
- **Out of scope(本仕様で扱わない)**
  - 同一プロバイダ種の複数登録(例: OpenAI 設定を本番用・検証用で 2 つ持つこと)
  - プロバイダ設定領域の動的な追加・削除(対応 4 種の固定スロットとする)
  - 保存済み API キーの消去(クリア)操作(キーは上書きのみ)
  - 対応プロバイダ種の追加(新ベンダー対応)
  - レガシー OpenAI 系統設定(`openai:*`)に依存する機能(ページパス提案 suggest-path 等)の統合・変更
  - 実行時の外部ネットワーク通信によるモデル一覧取得(ai-settings-model-picker の非スコープを踏襲)
  - 旧設定から新設定への自動移行(migration)
- **Adjacent expectations(隣接する前提)**
  - 許可モデル入力のカタログ選択方式(カタログ対応プロバイダは選択式、非対応プロバイダは自由入力。同梱カタログ + opt-in リフレッシュによる実効カタログ)は ai-settings-model-picker が提供する前提であり、本仕様はそれを所属プロバイダ単位で適用する。カタログのリフレッシュ機構(起動時/cron/手動)とその設定・UI 導線は picker の所管のまま維持され、本仕様の管理画面再構成で失われない。
  - 「許可モデル集合 = 認可境界。クライアントからのモデル指定はサーバが検証する」という原則は維持され、本仕様はその判定単位を(プロバイダ, モデル識別子)の組に拡張する。
  - モデルごとの provider オプション・メッセージ単位のモデル適用・ユーザー個人設定としての選択永続化の各挙動(mastra-multi-model-chat)は維持される。
  - 管理画面のレイアウト・操作構造は [ui-design/](./ui-design/) の Claude Design モック(AI Settings Multi-Provider / ProviderPanel)を参照とする(細部の実装表現は design で確定する)。

## Requirements

### Requirement 1: 複数プロバイダの構成(管理者)
**Objective:** 管理者として、複数の LLM プロバイダの接続設定を同時に構成し有効/無効を切り替えたい。そうすればアプリ全体を単一ベンダーに固定せず、用途に応じたモデル提供ができる。

#### Acceptance Criteria
1. The AI 設定機能 shall 対応プロバイダ(OpenAI / Anthropic / Google / Azure OpenAI)それぞれに固定の設定領域を提供し、複数プロバイダを同時に構成できるようにする(プロバイダの動的な追加・削除は提供しない)。
2. The AI 設定機能 shall 各プロバイダ種の構成を 1 つに限定する(同一プロバイダ種の多重構成は提供しない)。
3. When 管理者があるプロバイダの接続設定(資格情報を含む)を更新して保存する, the AI 設定機能 shall その設定を他のプロバイダの設定から独立して永続化する(他プロバイダの設定に影響を与えない)。
4. When 管理者が API キーを設定済みのプロバイダについて、新しいキーを入力せずに他の項目を更新して保存する, the AI 設定機能 shall 保存済みの API キーを維持する(保存済みキーの消去操作は提供しない)。
5. The AI 設定機能 shall 各プロバイダに有効/無効の切り替えを提供する。
6. When 管理者があるプロバイダを無効に切り替えて保存する, the AI 設定機能 shall そのプロバイダの接続設定・資格情報・許可モデルの設定を保持したまま、そのプロバイダに属する許可モデルをエンドユーザーの選択肢から除外する。
7. When 管理者が無効のプロバイダを有効に切り替えて保存する and そのプロバイダが構成済みである, the AI 設定機能 shall そのプロバイダに属する許可モデルをエンドユーザーの選択肢に復帰させる。
8. While あるプロバイダの API キーが設定済み, the AI 設定機能 shall キーの値そのものを表示せず、設定済みであることのみを示す。
9. The AI 設定機能 shall API キー等の秘匿値を API 応答およびログに含めない。
10. Where 対象プロバイダが Azure OpenAI, the AI 設定機能 shall 従来どおりの接続設定(リソース名または base URL、任意の API バージョン、Entra ID 認証の選択)を受け付ける。

### Requirement 2: プロバイダ横断の許可モデル集合(管理者)
**Objective:** 管理者として、登録した各プロバイダ配下のモデルを 1 つの許可モデル集合として統制したい。そうすればエンドユーザーに提供するモデルをプロバイダ横断で一元管理できる。

#### Acceptance Criteria
1. The AI 設定機能 shall 各許可モデルを「所属プロバイダ + モデル識別子」の組として管理する。
2. When 管理者が許可モデルを追加する, the AI 設定機能 shall 対象プロバイダの設定領域内での追加操作により所属プロバイダを一意に確定させる。
3. The AI 設定機能 shall 異なるプロバイダ配下であれば同一のモデル識別子を持つ許可モデルの共存を許可する。
4. If 同一プロバイダ配下に同一のモデル識別子が重複して登録される, then the AI 設定機能 shall 保存を拒否し、該当箇所にエラーを表示する。
5. If 対応プロバイダ種のいずれでもないプロバイダを所属プロバイダとする許可モデルが指定される, then the AI 設定機能 shall 保存を拒否する。
6. Where 所属プロバイダがモデルカタログ対応, the AI 設定機能 shall そのプロバイダにスコープしたカタログからの選択によりモデル識別子を入力させる。
7. Where 所属プロバイダがモデルカタログ非対応(例: Azure OpenAI), the AI 設定機能 shall モデル識別子の自由入力を受け付ける。
8. The AI 設定機能 shall 各許可モデルに対する provider オプションの設定を従来どおり提供する。
9. The AI 設定機能 shall 構成が未完了(資格情報未設定等)のプロバイダに対しても許可モデルの編集を許可する(そのモデルの利用可否は Requirement 6 の縮退規則に従う)。

### Requirement 3: グローバル既定モデル(管理者)
**Objective:** 管理者として、プロバイダ横断の許可モデル集合から既定モデルを 1 つ指定したい。そうすればエンドユーザーの初期体験を統制できる。

#### Acceptance Criteria
1. While 許可モデル集合に 1 件以上のモデルが存在する, the AI 設定機能 shall 許可モデル集合全体(全プロバイダ横断)でちょうど 1 つの既定モデルを保持する。
2. If 許可モデル集合に 1 件以上のモデルが存在する状態で、既定モデルが 0 個または 2 個以上のまま保存される, then the AI 設定機能 shall 保存を拒否し、許可モデル集合内から既定をちょうど 1 つ選ぶよう促す。
3. The AI 設定機能 shall 空の許可モデル集合(0 件)の保存を許可する(段階的セットアップ・全モデル撤去を可能にするため。その状態の AI 機能の可用性は Requirement 6 に従い「未設定」として扱う)。

### Requirement 4: チャットでのプロバイダ横断モデル選択(エンドユーザー)
**Objective:** エンドユーザーとして、チャット画面でプロバイダをまたいでモデルを選びたい。そうすれば質問の用途に応じて最適なベンダーのモデルを使い分けられる。

#### Acceptance Criteria
1. While チャットが利用可能, the チャット機能 shall すべての有効なプロバイダの許可モデルを、単一の選択操作で横断的に選べるモデルセレクタとして提示する。
2. The チャット機能 shall モデルセレクタの各選択肢がどのプロバイダに属するかを判別できる形で表示する。
3. When ユーザーがあるプロバイダのモデルを選択してメッセージを送信する, the チャットサーバ shall そのメッセージの応答生成に、選択された所属プロバイダのそのモデルを使用する。
4. When ユーザーがモデルセレクタでモデルを選択する, the チャット機能 shall その選択を所属プロバイダまで一意に特定できる形でユーザー個人設定として永続化し、次回以降の初期選択に用いる。
5. If 永続化されたユーザーの選択モデルがエンドユーザーの選択肢に含まれない(プロバイダの無効化・構成不備やモデルの除外を含む), then the チャット機能 shall 既定モデルにフォールバックする。
6. The チャットサーバ shall クライアントから指定されたモデルを有効なプロバイダの許可モデル(所属プロバイダとモデル識別子の組)に対して検証し、選択肢外の指定は既定モデルへ丸める。
7. The チャット機能 shall メッセージ単位のモデル適用およびスレッド途中でのモデル切替の既存挙動を、プロバイダをまたぐ切替でも維持する。

### Requirement 5: 環境変数による構成(運用者)
**Objective:** 運用者として、複数プロバイダの構成を環境変数のみでも設定したい。そうすれば構成管理を IaC・デプロイ設定に一元化できる。

#### Acceptance Criteria
1. The AI 設定機能 shall 複数プロバイダの接続設定(資格情報・有効/無効の状態を含む)および許可モデル集合を、環境変数のみで構成できるようにする。
2. While env-only モード(環境変数のみで設定)が有効, the AI 設定機能 shall プロバイダの接続設定(API キー等の資格情報・接続先設定・有効/無効の状態)を環境変数からのみ受け付け、管理画面では読み取り専用にする。
3. While env-only モードが有効, the AI 設定機能 shall 許可モデルの設定(所属プロバイダの指定・モデル識別子・provider オプション・既定モデル)の管理画面からの編集を引き続き許可する。
4. While env-only モードが有効, when 管理者が管理画面から許可モデルを編集して保存する, the AI 設定機能 shall 通常時と同一の検証(所属プロバイダの妥当性・同一プロバイダ内の重複禁止・既定モデルちょうど 1 つ)を適用する。

### Requirement 6: 設定不備時の部分縮退(運用者)
**Objective:** 運用者として、一部プロバイダの設定不備が AI 機能全体を止めないようにしたい。そうすれば可用性を保ちながら段階的に設定を整備できる。

#### Acceptance Criteria
1. If 有効に切り替えられたプロバイダに必須の接続設定の不備がある, then the AI サービス shall そのプロバイダに属する許可モデルをエンドユーザーの選択肢から除外し、不備の内容をログに出力する。
2. While 有効なプロバイダが 1 つ以上存在し、その配下に許可モデルが 1 つ以上ある, the AI サービス shall チャット機能を利用可能とする。
3. If 有効なプロバイダが 1 つも存在しない、または有効なプロバイダ配下の許可モデルが 1 つも存在しない, then the AI サービス shall AI 機能を未設定として扱い(チャット機能を無効化し)、アプリケーション本体の動作は継続させる。
4. If 既定モデルが無効なプロバイダ(無効切替または構成不備)に属している, then the チャット機能 shall 有効な許可モデルの中から代替の初期選択を決定し、チャットを利用可能に保つ。

### Requirement 7: 旧単一プロバイダ設定の置換(運用者・開発者)
**Objective:** 運用者・開発者として、旧来の単一プロバイダ設定と新しい複数プロバイダ設定が二重に存在しない状態にしたい。そうすれば設定体系の混乱と保守負担を避けられる。

#### Acceptance Criteria
1. The システム shall 旧来の単一プロバイダ設定(単一のプロバイダ選択・全プロバイダ共用の単一 API キー・単一の Azure 専用接続設定)を複数プロバイダ設定へ置き換え、旧設定項目を提供しない。
2. The システム shall 旧設定から新設定への自動移行を提供しない(プレリリース段階の置換として扱う)。
3. If 旧形式の設定値のみが存在し、新形式の複数プロバイダ設定が存在しない, then the AI サービス shall AI 機能を未設定として扱う。
