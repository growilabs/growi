# Search Filter Implementation Rules
- Brand new source code files specific to search filters must be created in `apps/app/src/features/search`. (Modifying existing shared server/service files to integrate these filters is permitted).
- Ensure all filters are compatible with the existing `SearchService` pipeline (`parseQueryString` → `resolveFilterData` → `appendCriteriaForQueryString`).
- The `editor:` filter must NOT resolve via MongoDB. It maps directly to the indexed `last_update_username` keyword field as an exact `term` match, symmetric with `author:` on `username`. The field is populated by the indexing pipeline (`aggregate-to-index.ts` + `prepareBodyForCreate`) and requires a full index rebuild to take effect on existing pages.
- `group:` resolution is scoped to the requesting user's **own** groups: fetch them with `UserGroup.find({ _id: { $in: userGroups } })` + the `ExternalUserGroup` equivalent (select `_id name`), build a `name → [groupId]` map (read `group.id`), and resolve typed names against it. There is **no** global `findOne({ name })` lookup and **no** member-user resolution. Because the lookup is scoped by id, a group the user does not belong to is absent from the map and resolves to nothing — membership is enforced structurally, with no separate intersect step. (This also dissolves the external-group `{name, provider}` name-uniqueness concern: only the user's own groups are ever seen, and same-name groups all resolve.)
- `resolveFilterData` must **early-return** (issuing zero MongoDB queries) when the user is a guest (`userGroups == null`) OR both `terms.group` and `terms.not_group` are empty. Guard on array emptiness (`.length === 0`), **not** `== null` — the parser initializes these arrays to `[]`, so a `== null` guard never fires and would run the group lookups on every search.
- The `group:` filter must use `{ terms: { granted_groups: [groupId, ...] } }` as its Elasticsearch clause — do not resolve to member usernames.
- The **positive** `group:` clause must be gated on whether the operator was typed (`terms.group.length > 0`), NOT on whether resolution produced IDs (`groupIds.length > 0`). When a typed `group:` resolves to no IDs (unknown or non-member group), still push `{ terms: { granted_groups: [] } }` — an empty `terms` array matches nothing, yielding 0 results; skipping the clause would drop the filter and return everything. This requires the clause builder to receive `terms`, not only the resolved data — a builder that takes resolved data alone cannot distinguish "not typed" from "typed but resolved empty" and will get this wrong.
- The **negation** `-group:` clause is the opposite: skip it when `notGroupIds` is empty (an unknown/non-member negated group must exclude nobody). A user must never be able to filter by a group they are not a member of.
