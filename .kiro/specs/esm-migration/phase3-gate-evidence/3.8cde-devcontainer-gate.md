## 3.8.c auth middleware snapshot diff + 認可マトリクス diff — PASS

### (1) route-middleware snapshot (`/tmp/route-mw-fresh.json` vs route-middleware-baseline.json)

- 273 apiv3 leaf endpoints / entries 構造一致。`entries[]` diff = **3 行のみ**:

```
POST /_api/v3/slack-integration/commands       [addSigningSecretToReq, S→$, checkCommandsPermission, <anonymous>]
POST /_api/v3/slack-integration/events         [verifyUrlMiddleware, addSigningSecretToReq, S→$, <anonymous>]
POST /_api/v3/slack-integration/interactions   [addSigningSecretToReq, S→$, parseSlackInteractionRequest, checkInteractionsPermission, <anonymous>]
```

- 差分は `addSigningSecretToReq` の直後の `verifySlackRequest` middleware の `.name` が
  `"S"` → `"$"` に変化した 1 文字のみ。順序・個数・前後の named guard はすべて完全一致。
- **根本原因**: `verifySlackRequest` は `@growi/slack/dist/middlewares` の minify 済み export
  (`packages/slack/dist/middlewares/verify-slack-request.js` が `$ as verifySlackRequest`)。
  baseline 取得時 (R.6.2) の `@growi/slack` ビルドでは内部名 `S` だったが、再ビルドで minifier の
  単一文字割り当てが `S`→`$` にシフトしただけ。**apps/app の ESM 移行とは無関係の外部パッケージ
  dist の表記揺れ**で、認可チェーン構造は不変 = guard 欠落 / 順序変化 / 未知 middleware 挿入なし。
- 無名関数: `<anonymous>` は terminal route body のみ (Implementation Note 0.3 と同運用)。新規無名 0 件。
- baseline ファイルは書き換えない (移行前スナップショット保全 + minify 名は再ビルドで揺れるため固定無意味)。

### (2) authz-matrix (`/tmp/authz-fresh.json` vs authz-matrix-baseline.json)

- 264 rows × 4 persona / vault-manager 非稼働 (ECONNREFUSED :3001) = baseline と同条件。
- 認可ゲート列 (unauthenticated / guest / readonly) は **264 行全件で完全一致** (admin 除外 diff = 0)。
- `matrix[]` diff = **1 行のみ**:

```
POST /_api/v3/import/upload
  baseline: {unauthenticated:403, guest:403, readonly:302, admin:599}
  fresh:    {unauthenticated:403, guest:403, readonly:302, admin:400}
```

- `599 = STATUS_UNCAUGHT_EXCEPTION` (capture ツールの sentinel: 「handler が応答ステータス割当て前に
  クラッシュ / 5s タイムアウト」)。baseline 取得環境では admin リクエストが post-auth でクラッシュ/timeout、
  fresh では正常な 400 (入力検証応答)。admin は認証通過済み persona なので **認可チェーンではなく
  handler のエラーハンドリング挙動の差 (599 クラッシュ→400 正常応答の改善方向)**。認可バイパスのリスクなし。
