## 経緯 (修正により green 化)

本 gate の実行で、type:module 化 (task 3.6) 以降 ESM 非対応のまま残っていた `__dirname` が
2 系統で顕在化し、修正した:

1. **Playwright ドメイン** (commit `f427a6f836`): `playwright.config.ts` を Node 24 がネイティブ ESM として
   読み込み `__dirname is not defined in ES module scope` で config ロードに失敗 → 直前の run #27550932498
   (sha `fb9a76bce2`) は全 12 E2E ジョブが test 実行前に一様 fail。`import.meta.dirname` 化で解消。
   (config + `playwright/utils/Login.ts` + `playwright/23-editor/with-navigation.spec.ts`)
2. **サーバ feature** (commit `ea3b744307`): `src/features/openai/.../convert-markdown-to-html.ts` と
   `src/features/page-bulk-export/.../export-pages-to-fs-async.ts` の `dynamicImport(pkg, __dirname)`。
   `server:ci` で落ちなかったのは Prisma 生成クライアントが `globalThis['__dirname']` をシムしていた
   (`src/generated/prisma/client.ts`) ためで、解決ベースが本来のディレクトリでなく脆い状態だった。
   task 3.5 のスイープは `src/server` のみが対象で `src/features/*/server` を取りこぼしていた。

## flaky 記録 (ESM 無関係)

- `run-playwright (firefox, 1/2, 6.0)` が初回 fail → `gh run rerun --failed` で green。
- 失敗テスト: `playwright/20-basic-features/comments.spec.ts`
  (`strict mode violation: locator('.page-comment-body') resolved to 2 elements`)。
- 原因は **test-isolation flaky**: 同テストは chromium / webkit / firefox-8.0 では緑。Playwright の retry が
  同一 mongo コンテナにコメントを重複追加し strict-mode 違反が累積するパターン。ESM 移行とは無関係の
  既存 test-quality 課題 (別件として切り出し推奨)。
