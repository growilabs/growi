# Requirements Document

## Project Description (Input)

### 誰が / 現状 / 変えたいこと
- **誰が**: GROWI の管理者（admin の「AI Settings」画面で LLM を設定する運用者）。
- **現状**: チャットで許可するモデルを、`AllowedModelsField` で **modelId の自由入力**として1件ずつ登録している。このため (1) 存在しない/綴り間違いの modelId を登録し得る、(2) 利用可能なモデル名を管理者が別途調べる必要がある、という課題がある。
- **変えたいこと**: この入力を、カタログを持つプロバイダでは **選択のみ（手入力不可）** にする。選択肢は **models.dev のカタログをリリース前段の取り込みステップで取得し、chat＋ツール対応モデルだけに絞ってコミットした同梱アセット**から、現在設定中のプロバイダ（`ai:provider`）にスコープして取得する。カタログを持たないプロバイダ（Azure OpenAI 等）は従来どおり自由入力を残す。

### 外部通信に関する前提
- **モデル一覧の提供（実行時の read パス）は外部通信を行わない**。カタログの基線は models.dev の api.json を **取り込みステップ（リリース前段の独立ステップ。手動実行も可）** で取得し、chat＋ツール対応モデルだけに生成時フィルタした**コミット済みの静的アセット（同梱カタログ）**である。一覧提供はローカルに保存されたカタログ（同梱カタログ、または後述の更新済みカタログ）を read するだけで、**リクエスト都度の外部取得は行わない**（自己ホスト/オフライン/エアギャップでも既定のまま機能する）。
- **カタログの更新（リフレッシュ）は一覧提供とは別の操作**として提供する: (a) **管理画面からの手動更新**（GROWI.cloud 等、同一バージョンを長く使う運用向け）、(b) **起動時更新オプション**（growi-docker-compose 等、焼き込み済みイメージ運用向け）、(c) **定期自動更新**。いずれの更新経路も **AI 機能が有効（`app:aiEnabled`）な場合に限り**作動し、その時に限りサーバが models.dev へ fetch してカタログを更新・永続化する。AI 機能は既定で無効のため **既定構成の GROWI は外部通信ゼロ**（エアギャップ環境は既定のまま影響なし）。起動時更新は既定 OFF、定期自動更新は既定で日次スケジュール（AI 有効化時に自動最新化。スケジュールを空文字にすれば無効化可）。
- models.dev の **runtime fetch（モデルルーター）は不採用**。推論は models.dev 経由ではなく native `@ai-sdk/*` で実行する（不変。カタログのリフレッシュは選択肢一覧の更新に限られ、推論経路には関与しない）。
- カタログには models.dev の権威的フィールド（`tool_call`・入出力 modality）が含まれ、chat＋ツール対応の判定に用いる（名前 heuristic に依存しない）。

### スコープ / 非スコープ
- **変えるのは**「admin が modelId をどう入力するか」と「そのカタログをどう最新化できるか」。カタログを持つプロバイダ = 選択のみ、カタログ無し = 自由入力。
- **カタログ更新手段を提供する**: 管理画面からの手動更新／起動時更新オプション／定期自動更新（いずれも AI 機能が有効な場合のみ作動。AI 既定 OFF のため既定構成は外部通信ゼロ）。
- **不変**: 許可リスト（`ai:allowedModels`）＝認可境界、既定モデル（`isDefault`）、chat 側の `PromptInputModelSelect`、推論の native `@ai-sdk/*` 実装。
- カタログの read・更新は **サーバ側のみ**で実行し、client には `{id,name}` の配列（`SelectableModel[]`）のみ返す（同梱カタログや取り込み処理を client バンドルに引き込まない）。

### 既知の制約（設計で扱う）
- **`azure-openai` は静的カタログに存在しない**（Azure のモデル ID は運用者定義のデプロイ名で列挙不可）。→ azure（およびカタログを持たないプロバイダ）は自由入力を残し、カタログを持つプロバイダは選択のみ（手入力不可）。
- **選択のみの帰結**: 新モデルは**カタログが更新されるまで**選べない。同梱カタログは基線であり、リリース時の再取り込み（vendoring）に加えて、**リフレッシュ（管理画面手動／起動時／定期）で運用中にも更新できる**。リフレッシュを使わない（使えない）環境では同梱カタログのままとなる（許容済み）。
- 絞り込みは models.dev の `tool_call`＋出力 modality を用いた**権威的判定**（chat＋ツール対応のみを提示）。名前パターンの推測には依存しない。リフレッシュ経路も**同一のフィルタ・検証**を適用する。
- カタログ内容は取り込んだ models.dev スナップショットのバージョンで drift する（provider→`{id,name}` 配列という shape は安定）。id と表示名 `name` は同一スナップショットから同時に取り込むため相互に drift しない（models.dev が `name` を持たないモデルは id をフォールバックとして格納）。
- 鮮度運用: 基線の取り込み（fetch＋フィルタ＋コミット）は**リリースビルドの前段の独立ステップ**で行い、リリースビルドはコミット済みアセットを read する（ビルド工程で fetch しない）。運用中の鮮度はリフレッシュ（手動／起動時／定期）で補う。

### 本 spec のスコープに含める既存 spec の更新（新 spec のタスクとして整合させる）
- **`mastra-multi-model-chat`**（`tasks-approved` / `ready_for_implementation=true`）: requirements の「モデル一覧取得 API は新設しない」前提の修正、design Non-Goals の整合。
- **`multi-llm-provider`**（`tasks-generated` / `ready_for_implementation=false`）: research D-2/D-3 の models.dev モデルルーター（runtime fetch）却下は維持しつつ、リリース前段で vendoring した静的カタログの read は別経路（実行時は通信なし）であり推論は native 実装のままである旨の注記を追加。

## Introduction

本機能は、GROWI 管理画面の「AI Settings」において、チャットで許可するモデル（許可リスト）を登録する際の **modelId 入力方式を変更**するものである。静的カタログを持つプロバイダ（OpenAI / Anthropic / Google）では、現在設定中のプロバイダにスコープした一覧からの **選択のみ（手入力不可）** とし、綴り間違いや存在しない ID の登録を防ぐ。選択肢の基線は製品に同梱された静的なモデルカタログ（models.dev をリリース前段の取り込みステップで取得し、chat＋ツール対応に絞ってコミットしたアセット）であり、**一覧提供（read パス）は外部ネットワーク通信を一切伴わず**、自己ホスト・オフライン・エアギャップ環境でも既定のまま機能する。

同梱カタログは公式イメージに焼き込まれた後は変化しないため、**カタログの更新（リフレッシュ）手段**を別途提供する: 管理画面からの手動更新、起動時更新オプション（growi-docker-compose 等の焼き込み済みイメージ運用向け）、および定期自動更新である。いずれの更新経路も **AI 機能が有効（`app:aiEnabled`）な場合に限り**作動する。AI 機能は既定で無効のため、**既定構成の GROWI は外部通信を一切行わない**。起動時更新は既定 OFF（opt-in）、定期自動更新は既定で日次スケジュール（AI を有効化すると自動で最新化される。エアギャップでの AI 運用等はスケジュールを空文字にして無効化できる）。作動時に限りサーバが models.dev へ fetch し、同一のフィルタ・検証を経たカタログを永続化して以後の一覧提供に反映する。更新に失敗した場合は保存済みの last-good（更新済みカタログ、無ければ同梱カタログ）を維持し、機能を停止しない。

静的カタログを持たないプロバイダ（Azure OpenAI 等。ID は運用者定義のデプロイ名）や、一覧の取得に失敗した場合は、従来どおり自由入力を提供する。選択のみとした帰結として、カタログに未収録のモデル（新規リリース直後など）はカタログが更新（リリース時の再取り込み、またはリフレッシュ）されるまで選択できない。

本機能が変更するのは「管理者がモデル ID をどう入力するか」のみであり、許可リストによる認可、既定モデルの扱い、モデルごとのオプション設定、推論の実行方式、およびチャット画面側のモデル選択 UI は変更しない。あわせて、本機能の導入により記述が矛盾する既存スペック（`mastra-multi-model-chat` / `multi-llm-provider`）の該当箇所を、本スペックのスコープとして整合更新する。

## Boundary Context

- **In scope**:
  - AI Settings（管理画面）における許可モデルの入力方式の変更: カタログを持つプロバイダは選択のみ、カタログを持たないプロバイダは自由入力。
  - 現在設定中のプロバイダにスコープした、外部通信を伴わない選択可能モデル一覧の提供（read パス）。
  - **カタログの更新（リフレッシュ）手段の提供**: 管理画面からの手動更新、起動時更新オプション、定期自動更新（いずれも AI 機能が有効な場合のみ作動。起動時更新は既定 OFF、定期自動更新は既定で日次。AI 既定 OFF のため既定構成は外部通信ゼロ）。更新済みカタログの永続化と一覧提供への反映（解決は更新済み／同梱の新しい方。更新済みが無ければ同梱）。
  - カタログを持たないプロバイダ／一覧取得失敗時に対する自由入力の維持。
  - 記述が矛盾する既存スペック（`mastra-multi-model-chat` / `multi-llm-provider`）の整合更新。
- **Out of scope**:
  - チャット画面側のモデル選択 UI・体験の変更。
  - モデルの推論実行方式（プロバイダ実装・呼び出し経路）の変更。外部ゲートウェイ経由のモデルルーターの採用。
  - 許可リストによる認可モデルの再設計、既定モデル選定ロジック・オプション設定の意味変更。
  - 一覧提供（read パス）でのリクエスト都度の外部取得（提供は常にローカル保存済みカタログの read。外部通信はリフレッシュ操作に限る）。
  - models.dev 以外のソース（各ベンダー API 等）からのモデル一覧取得。
  - Azure OpenAI のデプロイ名の自動列挙。
- **Adjacent expectations**:
  - 既存の許可リスト検証、既定モデルの一意性、モデルごとのオプション設定、環境変数専用モードの読み取り専用挙動、AI 機能の有効判定は、本機能が依拠する前提としてそのまま維持され、本スペックはそれらの意味を変更しない。
  - 提示されるモデル一覧の網羅性・鮮度はローカル保存済みカタログ（同梱、または最後に成功したリフレッシュ結果）の内容に従う。リフレッシュを使わない環境では同梱カタログの鮮度に従う。
  - カタログを持たないプロバイダにおいては、モデル登録の妥当性は管理者の自由入力に依存する。

## Requirements

### Requirement 1: プロバイダ別カタログからの選択のみによる登録
**Objective:** 管理者として、カタログを持つプロバイダでは許可モデルを一覧からの選択のみで登録したい。存在しない/綴り間違いの modelId 登録を防ぎ、利用可能なモデルを別途調べる手間をなくすため。

#### Acceptance Criteria
1. When 管理者が許可モデルの登録欄を操作したとき, the AI設定管理画面 shall 現在設定されているプロバイダに対応する選択可能なモデルの一覧（各モデルの公式表示名 `name` を表示ラベル、`id` を保持値とする）を提示する。
2. When 管理者が一覧からモデルを選択したとき, the AI設定管理画面 shall 選択された modelId を許可リストのエントリとして登録する。
3. The AI設定管理画面 shall 一覧から選択された値を、既存の許可リストエントリと同一の形式（modelId 文字列）で保持する。
4. While 設定中のプロバイダに対応する選択可能モデル一覧が存在する場合, the AI設定管理画面 shall モデル ID の自由入力（一覧に無い値の手入力）を受け付けず、一覧からの選択のみを許可する。
5. If 既に保存済みの modelId が現在の選択可能モデル一覧に含まれない場合, then the AI設定管理画面 shall その値を選択済みの値として保持し、管理者の操作なしに変更・削除しない。

### Requirement 2: 外部通信を伴わないオフラインでの一覧提供（read パス）
**Objective:** 運用者として、モデル一覧の提供（read パス）で外部サービスへの通信が発生しないことを保証したい。自己ホスト・エアギャップ・オフライン環境でも安全かつ確実に機能させるため。

#### Acceptance Criteria
1. When 管理画面がモデル一覧を提供するとき, the モデル選択機能 shall 外部ネットワークへの通信を一切行わず、ローカルに保存されたカタログ（同梱カタログ、または保存済みの更新済みカタログ）のみを read する。
2. Where ネットワークが遮断された環境で運用される場合, the モデル選択機能 shall ローカルに保存されたカタログのみからモデル一覧を提供する。リフレッシュが未作動（既定構成＝AI 無効）でも、遮断によりリフレッシュが失敗しても、製品に同梱された静的カタログ（または最後に成功した更新済みカタログ）のみで機能する。
3. The モデル選択機能 shall モデル一覧の提供（read パス）にあたり、外部のモデルカタログ／ベンダー API を参照しない（外部通信は Requirement 9 のリフレッシュ操作に限る）。

### Requirement 3: カタログを持たないプロバイダ・取得失敗時の自由入力
**Objective:** 管理者として、静的カタログを持たないプロバイダ（Azure OpenAI 等）や一覧取得に失敗した場合でも、モデル（デプロイ名を含む）を登録できるようにしたい。カタログの有無にかかわらず設定を完了できるため。

#### Acceptance Criteria
1. Where 設定中のプロバイダに対応する選択可能モデル一覧が存在しない場合（例: Azure OpenAI）, the AI設定管理画面 shall 従来どおりモデル ID（デプロイ名）の自由入力を提供する。
2. If 設定中のプロバイダの選択可能モデル一覧を取得できない場合, then the AI設定管理画面 shall 自由入力にフォールバックし、保存操作をブロックしない。
3. The AI設定管理画面 shall 自由入力を、カタログを持たないプロバイダおよび一覧取得失敗時に限って提供し、カタログを持つプロバイダでは選択のみとする。

### Requirement 4: 許可リストのセマンティクスと既存挙動の不変性
**Objective:** 運用者として、入力方式の変更以外の挙動が一切変わらないことを保証したい。認可・既定モデル・オプション設定・推論・チャット側 UI の回帰を防ぐため。

#### Acceptance Criteria
1. The モデル選択機能 shall 許可リスト（許可モデルの集合）を引き続き認可境界として維持し、許可外モデルの利用を許さない。
2. When 管理者が許可モデルを登録・削除・既定指定・オプション設定するとき, the AI設定管理画面 shall 既存の許可リストのセマンティクス（既定は非空時に1件のみ 等）をそのまま適用する。
3. The モデル選択機能 shall モデルの推論実行方式、およびチャット画面でのモデル選択 UI を変更しない。
4. The モデル選択機能 shall AI 機能の有効判定（設定済みと見なす条件）の基準を変更しない。

### Requirement 5: プロバイダ切替時の一覧追従と未設定時の扱い
**Objective:** 管理者として、プロバイダを切り替えたら選択肢もそのプロバイダのものに更新されてほしい。誤ったプロバイダのモデルを登録しないため。

#### Acceptance Criteria
1. When 管理者が設定中のプロバイダを変更したとき, the AI設定管理画面 shall 提示するモデル一覧を、新しいプロバイダに対応する内容へ更新する。
2. While プロバイダが未設定または未選択の場合, the AI設定管理画面 shall モデル一覧を提示せず、自由入力を提供する。

### Requirement 6: チャット・ツール対応モデルへの限定
**Objective:** 管理者として、チャットで使えないモデル（埋め込み・画像生成等）や、GROWI エージェントが必須とするツール呼び出しに対応しないモデルが選択肢に出ないようにしたい。選択のみのため、一覧の質がそのまま登録可能なモデルの質になるため。

#### Acceptance Criteria
1. When 選択可能モデル一覧を提示するとき, the モデル選択機能 shall ツール呼び出しに対応し、かつテキスト出力に対応した（chat 用途の）モデルのみを提示する。
2. The モデル選択機能 shall この判定を、同梱カタログが持つ権威的なメタデータ（ツール呼び出し可否・出力 modality）に基づいて行い、名前パターンの推測に依存しない。

### Requirement 7: 秘匿情報の非漏洩・管理者認可・環境変数専用モードの尊重
**Objective:** 運用者として、一覧取得機能が秘匿情報を漏らさず、管理者のみが利用でき、環境変数専用モードの読み取り専用挙動を尊重することを保証したい。

#### Acceptance Criteria
1. The モデル選択機能 shall モデル一覧の応答に API キーやプロバイダ資格情報を一切含めず、モデルの id と公式表示名 `name`（いずれも非秘匿情報）のみを返す。
2. If 管理者権限を持たない要求者がモデル一覧を要求した場合, then the モデル選択機能 shall その要求を拒否する。
3. While AI 設定が環境変数専用モードで運用されている場合, the AI設定管理画面 shall 既存の読み取り専用挙動を維持し、選択 UI からの編集を許可しない。

### Requirement 8: 既存スペックとの整合更新（本スペックのスコープ）
**Objective:** スペック保守者として、本機能の導入に伴い矛盾する既存スペックの記述を更新し、ドキュメント間の一貫性を保ちたい。

#### Acceptance Criteria
1. When 本機能を実装するとき, the スペック保守プロセス shall `mastra-multi-model-chat` の要件・設計から「モデル一覧取得の仕組みは設けない／許可モデルは管理者が手入力する」という記述を、オフライン静的カタログに基づく選択方式の採用へ更新する。
2. When 本機能を実装するとき, the スペック保守プロセス shall `multi-llm-provider` の調査記述に対し、外部通信を伴うモデルルーターの不採用を維持しつつ、静的カタログの読み取り経路は別物であり推論は従来実装のままである旨の注記を追加する。
3. The スペック保守プロセス shall 上記更新の後、関連スペック間でモデル入力方式に関する矛盾記述が残らない状態にする。

### Requirement 9: カタログの更新（リフレッシュ）
**Objective:** 運用者・管理者として、同梱カタログが焼き込まれた後（公式 Docker イメージ運用・同一バージョンの長期運用）でも、イメージ更新なしに選択可能モデルのカタログを最新化したい。新モデルを許可リストに追加できるようにするため。

#### Acceptance Criteria
1. When 管理者が管理画面からカタログ更新を指示したとき, the モデル選択機能 shall models.dev からカタログを取得し、同梱カタログと同一のフィルタ（`tool_call`＋text 出力）と検証（スキーマ・各対象プロバイダ1件以上）を適用したうえで、更新済みカタログとして永続化し、以後の一覧提供に反映する。
2. Where AI 機能が有効（`app:aiEnabled`）かつ起動時更新オプションが有効化されている場合, the モデル選択機能 shall サーバ起動後にカタログ更新を試行する。この試行は起動をブロックせず、失敗してもサーバは正常に稼働を継続する。
3. Where AI 機能が有効（`app:aiEnabled`）かつ定期自動更新スケジュールが設定されている場合, the モデル選択機能 shall 指定されたスケジュールでカタログ更新を試行する。定期自動更新スケジュールは**既定で日次**（例 `0 4 * * *`）であり、AI 機能を有効化すると既定で自動最新化されるが、AI 機能が無効な間は作動しない（スケジュールを空文字に設定すれば無効化できる）。
4. If カタログ更新が失敗した場合（ネットワーク到達不可・スキーマ不一致・空結果）, then the モデル選択機能 shall 保存済みの last-good カタログ（更新済みカタログ、無ければ同梱カタログ）を維持し、一覧提供を停止しない。
5. The モデル選択機能 shall 一覧提供時のカタログ解決を「更新済みカタログ（保存済み）と同梱カタログのうち**新しい方**（同梱が厳密に新しい場合のみ同梱を優先。更新済みが無ければ同梱）」とする。イメージ更新でより新しい同梱カタログが配布された場合、古い更新済みスナップショットがそれを覆い隠さない。
6. The モデル選択機能 shall 起動時・定期のカタログ更新を **AI 機能が有効（`app:aiEnabled`）な場合に限り**実行する。AI 機能は既定で無効であるため、**既定構成の GROWI は外部通信を一切行わない**（既定構成のエアギャップ環境に影響を与えない）。起動時更新は既定 OFF（opt-in）、定期自動更新は既定で日次スケジュールだが、いずれも AI 機能が無効な間は作動しない。AI 機能を有効化した運用で外部通信を避けたい場合（エアギャップでの AI 利用等）は、定期自動更新スケジュールを空文字に設定して無効化できる。管理画面からの手動更新は管理者の明示操作に限る。取得先はビルトインの models.dev URL に固定する。
7. If 管理者権限を持たない要求者がカタログ更新を要求した場合, then the モデル選択機能 shall その要求を拒否する。取得先はビルトインの models.dev URL に固定し、要求者が取得先を指定することはできない。
