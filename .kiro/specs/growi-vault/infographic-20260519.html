Overview

GROWI Vault とは何か

ユーザーが自分の閲覧権限を持つ GROWI ページ群を git で取得できるようにする機能。GROWI.cloud の新サービスメニューとして提供する。

🖥️

標準 git クライアントで clone できる

追加ツール不要。git clone https://growi.cloud/_vault/repo.git だけで動く。認証は既存の Personal Access Token (PAT)。

📁

ページ階層がそのままディレクトリ構造に

GROWI の /A/B/C が clone 結果の A/B/C.md として現れる。AI エージェント・grep・Obsidian から直接参照できる。

🔐

ACL を git にも適用

公開/グループ/自分だけ の GROWI 権限がそのまま git 経由でも適用。他ユーザーの非公開ページは存在自体が観測不可能。

🔄

更新が自動で git に反映

GROWI でページを編集すると次回 git pull で差分が取得できる。rename・権限変更も自動伝播。

🔴 Before — 現状
GROWI ページ → MongoDB のみ
grep / find → ❌ 使えない
git clone → ❌ 使えない
AI エージェント → ❌ 使えない
↓
✅ After — GROWI Vault
GROWI ページ → git repo → ローカル .md
grep / find → ✓ 動作する
git clone/pull → ✓ 動作する
AI エージェント → ✓ 動作する
Architecture

システム構成

2 つのコンポーネントが協調する。セキュリティ境界は apps/app 一箇所 に集約し、vault-manager は外部から完全に隔離する。

ComposeViewController GitProxyController (upload-pack) VaultInstructionWatcher (change stream) VaultNamespaceBuilder (git objects) VaultViewComposer (per-user merge) VaultUploadPackSpawner VaultMaintenanceScheduler

Infrastructure

インフラ要件

GROWI.cloud で Vault を提供するために新たに必要になるリソースと制約。

🗄️

Filestore (GCP NFS)

bare repo の物理配置。POSIX semantics・atomic rename・真の file locking が必須。GCSFuse・S3 FUSE 等 object storage backed FS は非対応。

🚨 新規手配必須
☸️

StatefulSet (replicas=1)

vault-manager は single writer 前提。leader election 機構なし。k8s の StatefulSet で物理的に replicas=1 を保証する。Pod 再生成での不可時間は 1〜2 分 (MVP SLO 許容)。

新規 StatefulSet
🔒

NetworkPolicy

vault-manager への Ingress は apps/app Pod からのみ 許可。外部・他 namespace からの直接アクセスを NetworkPolicy で遮断する。認証境界が apps/app に集約されることで初めて成立する設計。

NetworkPolicy 追加
🐳

Dockerfile (vault-manager)

apps/app の distroless とは異なり dhi.io/node:24-debian13-dev を使用。git upload-pack spawn に git binary v2.30+ が必要なため。5 stage + turbo prune 構成は apps/app と揃える。

新規 image
🔑

k8s Secret (shared secret)

apps/app と vault-manager の service-to-service 認証に使う VAULT_MANAGER_INTERNAL_SECRET を k8s Secret として両 Pod に注入。DB への保存禁止。ローテーション時は両 Pod を再起動。

k8s Secret
🍃

MongoDB (既存 replica set)

新規コレクション 4 つ (vault_instructions, vault_namespace_state, vault_user_views, vault_sync_state) を追加。replica set 必須は既存要件。change stream 用の resume token 保持に対応済み。

既存 replica set を共用
ストレージ環境別まとめ: dev/docker-compose → local bind mount、 self-host 単一 Pod → local fs (SSD 推奨)、 self-host multi Pod → NFSv4 / Filestore / EFS、 GROWI Cloud → Filestore (GCP managed NFS) 必須。
Configuration

環境変数・設定

デプロイ時に設定が必要な環境変数と、管理者が行う初期設定。

環境変数 設定先 Pod 必須 / 任意 説明
VAULT_ENABLED apps/app 任意 機能 ON/OFF の初期値。管理 UI でも変更可能 (app:vaultEnabled)
VAULT_MANAGER_ENDPOINT apps/app 必須 vault-manager の内部 URL (例: http://vault-manager:3001)。DB 保存禁止・env var only。
VAULT_MANAGER_INTERNAL_SECRET apps/app + vault-manager 必須 service-to-service 認証の shared secret。k8s Secret で両 Pod に同値を注入。DB 保存禁止。
VAULT_REPO_PATH vault-manager 必須 bare repo の物理パス。Filestore の mount point 配下に設定する。
VAULT_SQUASH_COMMIT_THRESHOLD vault-manager 任意 squash トリガーの commit 数閾値 (デフォルト 1000)
VAULT_GC_INTERVAL_HOURS vault-manager 任意 git gc の実行間隔 (デフォルト 24h)
⚠️ シークレット管理: VAULT_MANAGER_ENDPOINT と VAULT_MANAGER_INTERNAL_SECRET は env var only で DB に保存されない。 k8s Secret として管理し、ローテーション時は apps/app・vault-manager 両 Pod を再起動すること。
Deployment

デプロイ手順

GROWI.cloud に Vault を追加するための作業ステップ。

1

Filestore (GCP NFS) のプロビジョニング

vault-manager が bare repo を置く共有 NFS を用意する。POSIX semantics と atomic rename が保証された Filestore を選択する。GCSFuse は非対応なので注意。

新規リソース GCP Filestore
2

vault-manager コンテナ image のビルドと push

apps/growi-vault-manager/Dockerfile (Ts.ED + git binary 同梱) をビルドして registry に push する。base image は dhi.io/node:24-debian13-dev。

新規 image 5 stage build
3

k8s Secret の作成 (VAULT_MANAGER_INTERNAL_SECRET)

apps/app と vault-manager の両 Pod に同じ secret 値を注入する k8s Secret を作成する。

k8s Secret
4

vault-manager の StatefulSet・Service・NetworkPolicy を適用

StatefulSet (replicas=1)・内部向け Service・Filestore の PersistentVolumeClaim・NetworkPolicy (apps/app Pod からの Ingress のみ許可) を apply する。

新規 StatefulSet NetworkPolicy PVC (Filestore)
5

apps/app の環境変数を設定してデプロイ

VAULT_MANAGER_ENDPOINT と VAULT_MANAGER_INTERNAL_SECRET を apps/app の環境変数に追加し、再デプロイする。

環境変数追加 既存 apps/app を更新
6

管理 UI から Vault を有効化 → Bootstrap 実行

GROWI 管理画面の Vault 設定で Feature Toggle を ON にし、「Prepare GROWI Vault」ボタンを押して bootstrap を開始する。全ページが vault に取り込まれるまで clone/fetch は 503 で待機状態になる。進捗バーで確認可能。

管理者操作 10K ページ → 約 10 分
7

動作確認

PAT を使って git clone https://<growi-host>/_vault/repo.git を実行し、自分が閲覧権限を持つページが正しく clone されることを確認する。/_vault/repo.git/git-receive-pack が 403 を返すことも確認する。

Observability

監視・観測

運用時に確認できるエンドポイントと管理画面の観測ポイント。

💚 ヘルスチェック

vault-manager の MongoDB 接続・change stream・bare repo 到達性を確認する。

GET /health

正常: 200 {"status":"ok"} / 異常: 503 + details

📊 ストレージ統計

namespace 数・commit 数・loose object 数・repo サイズ・最終 squash/gc 時刻。管理 UI が表示する。

GET /internal/storage-stats

shared secret Bearer 認証。admin UI が 5〜10 秒でポーリング。

🔄 Bootstrap 進捗

初回有効化時の全ページ取り込み進捗。管理 UI の進捗バーで確認。

vault_sync_state.bootstrapState

pending / running / done / failed。done になるまで clone は 503。

📝 監査ログ

clone/fetch 完了・認証失敗をタイムスタンプ・ユーザー・操作種別付きで既存 audit log に記録。

既存 audit log collection

vault.clone-complete / vault.auth-failure でフィルタ可能。

⚡ Instruction 処理遅延

vault_instructions の未処理件数 (processedAt: null) がページ編集後に積み上がっていないかを確認。

vault_instructions

TTL 86400 秒で自動削除。retry 上限超過時は lastError フィールドに記録。

🔧 Maintenance ログ

squash (1000 commit 超 or 1h 経過) と git gc (24h ごと) の実行結果をログに記録。

@growi/logger (pino)

所要時間・対象 namespace 数・before/after object 数を出力。

⚠️ 災害復旧: bare repo が全消失した場合でも管理 UI から bootstrap を再実行 するだけで完全復旧できる。 vault-manager は instruction を受け取って再処理するのみで、bootstrap は apps/app が全ページを再度 outbox に投入する。
