Overview

GROWI Vault とは何か

ユーザーが自分の閲覧権限を持つ GROWI ページ群を git で取得できるようにする機能。GROWI.cloud のサービスメニューとして提供する。

🖥️

標準 git クライアントで clone できる

追加ツール不要。git clone https://growi.cloud/_vault/repo.git だけで動く。認証は既存の Personal Access Token (PAT)。

📁

ページ階層がそのままディレクトリ構造に

GROWI の /A/B/C が clone 結果の A/B/C.md として現れる。AI エージェント・grep・Obsidian から直接参照できる。

🔐

ACL を git にも適用

公開 / グループ / 自分だけ の GROWI 権限がそのまま git 経由でも適用される。他ユーザーの非公開ページは存在自体が観測不可能。

🔄

更新が自動で git に反映

GROWI でページを編集すると次回 git pull で差分が取得できる。rename・権限変更も自動伝播。

🛡️

drift を自動検出して補修

pages.updatedAt watermark sweep で MongoDB と vault tree の乖離を検出。検出時は既存 instruction 経路で自動補修。bootstrap も resilient な state machine で再起動・失敗からの自動再開を保証。

🎯

UI から手動 reconcile も可能

admin は全範囲、一般ユーザーは自分が write 権を持つ sub-tree のみを対象に、PageTree や GrowiContextualSubNavigation からピンポイントで再同期をトリガーできる。watermark sweep が拾えない局所 drift の救済経路。

Architecture

システム構成

2 つのコンポーネントが協調する。セキュリティ境界は apps/app 一箇所 に集約し、vault-manager は外部から完全に隔離する。

Mechanism

仕組みの解説

MongoDB が唯一の真実の源 (SSOT)。vault は MongoDB の内容を git 形式に "投影" したものに過ぎない。bootstrap・通常更新・drift 自動補修・手動 reconcile の 4 経路がすべて同じ outbox を共有する。

1

管理者が Vault を有効化

admin UI で「Prepare GROWI Vault」ボタンを押す。または VAULT_BOOTSTRAP_ON_START=true / force で自動起動も可能。

2

state machine が遷移を管理

vault_sync_state.bootstrapState は pending → running → verifying → done (失敗時は failed / aborted / force-pending) の 7 状態。true で再起動しても done 状態は破壊されない。

vault_sync_state
3

cursor で全ページを stream + bulk-upsert を投入

published かつ /trash 以外の全ページを cursor で読み出し、各ページの ACL から namespace を判定。namespace 単位で bulk-upsert instruction を chunk size ごとに outbox へ insert。

VaultBootstrapper vault_instructions
4

vault-manager が change stream で受信して bare repo に書き込む

Markdown 本文 → blob → tree（ディレクトリ構造）→ commit → namespace ref を更新。isomorphic-git で実装。

VaultNamespaceBuilder bare repo
5

completeness 検証 → state: done

処理件数と推定全件数を比較する軽量検証を経て verifying → done に遷移。clone / fetch が解禁される。失敗時は failed で停留し、起動時 retry policy で有界 backoff 再試行 (上限超過は admin UI に escalate)。

vault_sync_state VaultResilienceLayer
🔄 通常運用 — ページ更新フロー
1

ユーザーがページを編集・保存

通常の GROWI 編集操作。ユーザーは Vault の存在を意識しない。

GROWI ユーザー
2

MongoDB に page + revision を永続化 (SSOT 更新)

PageService が MongoDB を更新する。これが唯一の正本更新。vault の更新はこの後に非同期で行われる。

MongoDB (SSOT)
3

PageService が in-process event を emit

pageEvent.emit('create' | 'update' | 'delete' | 'rename' | ...) を発火。

PageService
4

VaultDispatcher が namespace を計算して outbox に投入

ページの ACL から namespace を判定。ACL 変更の場合は旧 namespace に remove・新 namespace に upsert を発行。大量イベントは bulk-upsert に自動 coalesce (1s / 最大 1000 件)。

VaultDispatcher vault_instructions
5

vault-manager が change stream で検知して bare repo を更新

processedAt = null の instruction を冪等に処理。revisions.body を ID 指定で取得 → blob → tree → commit → namespace ref を更新。次回 git pull で差分が手元に反映される。

vault-manager bare repo

Correctness layer (発火源で 2 系統)

通常更新フローで取りこぼした drift を埋める仕組み。誰がトリガーするかで 2 系統に分離している。両者は同じ outbox 経路 + vault-manager の冪等性に乗るため、衝突しても最終状態は一意に収束する。

🛡️ 自動 (system-triggered) — drift detection
1

VaultResilienceLayer が定期 sweep を起動

config 設定の周期で起動。pages.updatedAt 順で前回 watermark 以降の差分を読み出す。O(N) 全件 scan は避ける軽量設計。

VaultResilienceLayer
2

vault tree との乖離を判定

編集 / trash 移動 / restore 由来の drift をカバー。path change / grant drop / hard delete drift は構造的に検出不能 (それらは手動 reconcile に委ねる)。

pages.updatedAt
3

既存 instruction op で補修発行 (新規 op は増やさない)

検出された乖離に対し bulk-upsert / remove を outbox に投入。以降は通常フローと同じ経路で vault-manager が処理する。

vault_instructions
4

watermark 進捗を保存 + admin UI に observability surface

vault_sync_state.driftDetector* に sweep の最終位置と統計を保存。admin UI の Drift Activity セクションに件数と最終 sweep 時刻を表示。

vault_sync_state admin UI
🎯 手動 (user-triggered) — targeted reconcile
1

admin または一般ユーザーが UI から起動

admin: /admin/vault の Reconcile セクションから sub-tree 指定。一般ユーザー: PageTree や GrowiContextualSubNavigation の dropdown から自分のページに対して 1 クリック。

admin user
2

VaultReconcileService の accept gate が判定

target validation → bootstrap 完了確認 → role 別の page 数上限 (admin / user) → per-user / system-wide 同時実行上限 → ACL 評価 (一般ユーザーは write 権を持つ範囲のみ) を順番に通過。findOne × 1 + 非 admin 時のみ getUserRelatedGroups × 1 の軽量 I/O。

VaultReconcileService vault_reconcile_log
3

非同期 orchestrator が cursor で stream + bulk-upsert を投入

受付完了後は background で eligible page を cursor stream し、namespace 単位の bulk-upsert を outbox へ投入。orchestrator 側で limit(plannedPageCount + 1) のハードキャップを再適用 (受付ゲート見積もり stale 時の防御層)。

ReconcileOrchestrator vault_instructions
4

履歴と監査ログに記録

受付 / 開始 / 完了 / 失敗 / 拒否 / partial-acl-filtered の 5 種を audit event として emit。vault_reconcile_log に進捗と最終状態を保存し、admin UI の history table に表示。

vault_reconcile_log audit log + history table
Infrastructure

インフラ要件

GROWI.cloud で Vault を提供するために新たに必要になるリソースと制約。

🗄️

Filestore (GCP NFS)

bare repo の物理配置。POSIX semantics・atomic rename・真の file locking が必須。GCSFuse・S3 FUSE 等 object storage backed FS は非対応。

🚨 新規手配必須
☸️

StatefulSet (replicas=1)

vault-manager は single writer 前提。leader election 機構なし。k8s の StatefulSet で物理的に replicas=1 を保証する。Pod 再生成での不可時間は 1〜2 分。multi-replica 化は将来 spec (growi-vault-ha) で扱う。

新規 StatefulSet
🔒

NetworkPolicy

vault-manager への Ingress は apps/app Pod からのみ 許可。外部・他 namespace からの直接アクセスを NetworkPolicy で遮断する。認証境界が apps/app に集約されることで初めて成立する設計。

NetworkPolicy 追加
🐳

Dockerfile (vault-manager)

apps/app の distroless とは異なり dhi.io/node:24-debian13-dev を使用。git upload-pack spawn に git binary v2.30+ が必要なため。5 stage + turbo prune 構成は apps/app と揃える。

新規 image
🔑

k8s Secret (shared secret)

apps/app と vault-manager の service-to-service 認証に使う VAULT_MANAGER_INTERNAL_SECRET を k8s Secret として両 Pod に注入。DB への保存禁止。ローテーション時は両 Pod を再起動。

k8s Secret
🍃

MongoDB (既存 replica set)

新規コレクション 5 つ (vault_instructions, vault_namespace_state, vault_user_views, vault_sync_state, vault_reconcile_log) を追加。replica set 必須は既存要件。change stream 用の resume token 保持に対応済み。

既存 replica set を共用
ストレージ環境別まとめ: dev/docker-compose → local bind mount、 self-host 単一 Pod → local fs (SSD 推奨)、 self-host multi Pod → NFSv4 / Filestore / EFS、 GROWI.cloud → Filestore (GCP managed NFS) 必須。
Configuration

環境変数・設定

デプロイ時に設定が必要な環境変数。Reconcile / Resilience の細かなパラメータ (page 数上限・concurrency・retention 等) は admin UI と configManager 経由で実行時に変更可能。

環境変数 設定先 Pod 必須 / 任意 説明
VAULT_ENABLED apps/app 任意 機能 ON/OFF の初期値。管理 UI でも変更可能 (app:vaultEnabled)
VAULT_BOOTSTRAP_ON_START apps/app 任意 true / force / 未設定。true で初回起動時のみ bootstrap、done 状態の再起動では何もしない (state machine による safe restart)。force は明示的に全 wipe + 再 bootstrap。
VAULT_MANAGER_ENDPOINT apps/app 必須 vault-manager の内部 URL (例: http://vault-manager:3001)。DB 保存禁止・env var only。
VAULT_MANAGER_INTERNAL_SECRET apps/app + vault-manager 必須 service-to-service 認証の shared secret。k8s Secret で両 Pod に同値を注入。DB 保存禁止。
VAULT_REPO_PATH vault-manager 必須 bare repo の物理パス。Filestore の mount point 配下に設定する。
VAULT_SQUASH_COMMIT_THRESHOLD vault-manager 任意 squash トリガーの commit 数閾値 (デフォルト 1000)
VAULT_GC_INTERVAL_HOURS vault-manager 任意 git gc の実行間隔 (デフォルト 24h)
⚠️ シークレット管理: VAULT_MANAGER_ENDPOINT と VAULT_MANAGER_INTERNAL_SECRET は env var only で DB に保存されない。 k8s Secret として管理し、ローテーション時は apps/app・vault-manager 両 Pod を再起動すること。
Reconcile / Drift / Retry の設定: role 別 page 数上限 (app:vaultReconcileMaxPagesPer{User,Admin}Request) / per-user および system-wide concurrency / chunk size / retention 日数 / drift sweep 間隔 / retry policy 等は configManager 配下のキーとして admin UI から実行時に変更可能。env var 経由の override も同名で受け付ける。
