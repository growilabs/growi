What's New Since 5/19

resilience と reconcile が解決したもの

「誰がトリガーするか」を境界として system-triggered (自動) と user-triggered (手動) の 2 spec に分離。両者は vault_instructions outbox を共有し、vault-manager の冪等性契約に乗る。

🛡️
resilience
System-triggered

「bootstrapState: 'done' でも実体の完全性は保証されない」「失敗が fire-and-forget」「VAULT_BOOTSTRAP_ON_START=true でデータ消失リスク」という MVP 時点の 5 つの構造的欠陥を解消し、System 側の correctness 保証層を確立した。

  • 7 状態の state union による bootstrap state machine
    pending / running / verifying / done / failed / aborted / force-pending を明示状態として定義。VAULT_BOOTSTRAP_ON_START=true 再起動で reset-all が走らない不変条件を仕様化。
  • watermark-based drift detector v1
    pages.updatedAt watermark sweep で編集 / trash 移動 / restore 由来の drift を検出。検出時は既存 bulk-upsert instruction 経路に再発火 (新規 op を増やさず)。O(N) 全件 scan を回避。
  • 起動時自動再試行 (bounded exponential backoff)
    前回 failed 状態で起動した場合、retry policy で resume を試行。limit 超過時に admin UI に force warning として surface し人手介入を求める escalation 経路。
  • heartbeat / instanceId / stale 正規化
    startup migration で前回 process の crash 残留 (running + instanceId なし) を failed: process-restarted に正規化。後続 reconcile spec が同じ pattern を pending 残留にも適用 (HistoryStore.normalizeStaleLifecycle)。
  • admin UI 拡張
    /admin/vault に Bootstrap state / Completion Reliability / Auto-Retry Status / Drift Activity / Force Warning Banner の 5 section を追加。完了状態の信頼性指標と自動補修活動を観測可能に。
🎯
reconcile
User-triggered

watermark drift detector が構造的に拾えない局所 drift (path change / grant drop / hard delete) を、ユーザーが UI から ACL-scoped に手動補修できる経路を提供。force bootstrap (全 wipe) と watermark sweep の間を埋めるピンポイント手段。

  • admin + 一般ユーザーの 2 経路
    admin は全範囲 / 一般ユーザーは自分が write 権を持つ sub-tree のみ。POST /vault/reconcile (admin) と POST /vault/page/reconcile (user) の 2 endpoint で実現。
  • accept gate の I/O 制約
    受付ゲート中の DB I/O を findOne × 1 + 非 admin 時のみ getUserRelatedGroups × 1 に固定。countDocuments 等の全 scan を発行しない設計で、accept gate p99 ≤ 200ms を達成。
  • defense-in-depth な hard cap
    受付ゲートは page.descendantCount stale なら見積もり違いを起こすため、orchestrator 側で limit(plannedPageCount + 1) のハードキャップを再適用。stale 検知時は failed: limit-exceeded で停止。
  • PageTree / GrowiContextualSubNavigation 統合
    一般ユーザーは自分のページの dropdown menu から 1 クリックで reconcile 起動可能。modal 内で path 確認 → submit → accepted feedback の simple flow。i18n は commons namespace で全ページ対応。
  • 冪等性に委ねた競合許容
    resilience の自動補修と reconcile の手動補修が同じ vault_instructions outbox を共有するため衝突しうるが、vault-manager の content-addressing 冪等性により最終状態が一意に収束する設計を両 spec で明示。
E2E Findings

Playwright 実走で発掘した 4 つの実バグ

Task 5.3 (admin/user UI E2E シナリオ検証) を実装して chromium で実走したところ、unit test では拾えていなかった production wiring / client 統合層 / i18n 配置 の defect が連続して見つかった。いずれも feature 境界内 (LOCAL) で修正し、upstream spec への影響はなし。「unit test だけでは production wiring の正しさは検証しきれない」という教訓が残った。

1
apiv3 prefix の重複
client component の apiEndpoint 値が /v3/vault/reconcile だったが、apiv3Post が自動で /_api/v3 を prepend するため、実 URL は /_api/v3/v3/vault/reconcile となり 404。
/v3/ prefix を削除 (admin / user 両 endpoint)
2
production wiring の脱漏
apiv3/index.js が createVaultAdminRouter を呼んでおり createVaultAdminRouterWithDeps (reconcileService singleton を注入する factory) を使っていなかった。/reconcile-history が 500、user route はそもそも mount されていない状態。
factory を WithDeps 版に切替 + user route mount を追加
3
Modal lifecycle leak (Dropdown 内)
PageReconcileMenuItem 内に useState で modal を保持していたが、親の Dropdown が outside-click で auto-close すると component ごと unmount され Modal も消える race condition。
Modal を親 (GrowiContextualSubNavigation + use-page-item-control) に lift up。PageDeleteModal / CreateTemplateModal と同 pattern
4
i18n namespace 配置ミス
reconcile reject reason の i18n key が admin.json 配下に置かれていたが、reconcile modal は user-facing page からも開くため admin namespace 未読込のページで生キー文字列が表示される。
growi-vault.reconcile.* を commons.json へ移動 + useTranslation('commons') へ変更
E2E の価値: 227 件の unit test と integration test が全 pass していたにも関わらず、上記 4 件は実 server を立てて UI を操作した時点で初めて顕在化した。Task 5.3 の追加コストは テストファイルそのもの + 4 件の bugfix という形で結果的に膨らんだが、それらは UI を実利用する前に発見されなかった以上、本来 production 障害として現れていたはずのもの。UI 機能を完成 (DONE) と宣言する直前の Playwright E2E は今後も必須。
Dependency Map

sub-spec 間の依存関係

各 sub-spec は vault_instructions outbox + 既存 instruction op を共有経路として、新規 op を増やさず冪等性契約を維持する設計。

growi-vault-gateway REFERENCE
PAT 認証 + ACL 評価 + read-only git smart HTTP / read API の owner。state を持たない (per-replica で stateless)。
provides: VaultPatAuth · VaultNamespaceMapper · admin UI 基盤
growi-vault-manager REFERENCE
instruction → git materialization の冪等な consumer。.git/index.lock 物理制約により同一 namespace への並行書き込み不能。
provides: idempotent instruction op handler · bare repo write owner
growi-vault-resilience DONE
gateway の bootstrap 経路と manager の reset-all op を再設計対象とし、置き換え設計を提示する。
depends on: gateway (interface) · manager (idempotency)
provides: bootstrap state machine · vault_instructions outbox write owner · heartbeat / instanceId primitive
growi-vault-reconcile DONE
outbox 経路を相乗りする consumer。新規 op を導入せず既存 bulk-upsert を再利用。
depends on: gateway (PAT/ACL interface) · resilience (instruction outbox + state machine 前提)
provides: user-triggered targeted reconcile · vault_reconcile_log
growi-vault-ha 将来予定 · 未着手
vault-manager の single-replica 前提を解消し、Competing Consumers + per-namespace lease で multi-replica 化。本 spec 適用後、resilience の drift detection は「observability 主、最小補修副」に縮小可能。
depends on: resilience (heartbeat / instanceId primitive 共有) · reconcile (multi-replica 下での sub-tree reconcile lease / claim を本 spec scope に含めるか要検討)
provides (予定): namespace_leases collection · vault_instructions atomic claim · failover (秒オーダー TTL)
境界分離の鍵: 「機能の重さ」ではなく「誰がトリガーするか」を境界として spec を切り出したことで、responsibility map が clean に保たれた。同じ vault_instructions outbox を共有しつつも、resilience は system-triggered の write owner、reconcile はその consumer として競合許容方針 (冪等性で最終収束) を両 spec で明示している。
